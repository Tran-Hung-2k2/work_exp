# Golang
  * DI (Dependency Injection)
    * loose coupling qua interface
    * inject dependency qua constructor (DI thủ công) -> phụ thuộc abstraction, không phụ thuộc implementation
  * Interface & Embedding
    * Go không có inheritance
    * implement đủ method → auto satisfy interface
    * embedding ≠ kế thừa
    * không có override → chỉ là method shadowing

# Concurrency (Go)  
  * goroutine nhẹ hơn thread OS, stack nhỏ (~2KB) và grow/shrink động (thread ~1MB–8MB cố định)
  * được quản lý bởi Go runtime (user-space), không qua OS kernel, context switch tại safe point → chỉ lưu state tối thiểu (PC, SP), không cần syscall, thread có thể bị interrupt bất kỳ → phải save full CPU context + switch kernel
  * M:N model (n goroutine chạy trên m thread)
  * race condition khi nhiều goroutine access shared data
  * tạo/huỷ cực nhanh → scale hàng chục nghìn goroutine

# Paging
  * Offset pagination
    * support jump page
    - O(n) với offset lớn (scan + skip) -> chậm
    - duplicate / missing khi data thay đổi
  * Cursor pagination
    * O(1) (index seek)
    * stable hơn offset
    * phù hợp infinite scroll / real-time feed
    - không jump page
    - có thể miss data mới insert (tuỳ direction)
  * Count:
    1. COUNT(*) = scan toàn bộ table (O(n)) (do MVCC → phải check visibility từng row)
    2. reltuples (estimate)
      * O(1), cực nhanh
      - phải ANALYZE để update
      - nhớ filter relkind='r'

# PostgreSQL
  * Storage (cách lưu data)
    - Data lưu theo page (default 8KB), 1 table = 1 hoặc nhiều file, mỗi file gồm nhiều page, DB đọc/ghi theo page
    - 1 page chứa nhiều tuple (row version), có header + pointer + data 

  * MVCC (multi-version)
    - Mỗi row có nhiều version (tuple), UPDATE/DELETE KHÔNG overwrite: → tạo row mới, row cũ vẫn tồn tại 
    - Mỗi row có:
      - xmin: transaction tạo
      - xmax: transaction xoá/update 
    - Query đọc theo snapshot → mỗi transaction thấy “view riêng”
    - Dead tuple sẽ được dọn bởi VACUUM

  * WAL (Write-Ahead Log)
    - WAL = log tuần tự (append-only) của mọi thay đổi 
    - Quy tắc: WAL phải ghi xuống disk TRƯỚC data page (write-ahead) 

  * Write flow
    1. Modify data trong shared buffer (RAM)
    2. Tạo WAL record (ghi thay đổi)
    3. Append vào WAL buffer
    4. COMMIT → fsync WAL xuống disk
    5. Data page flush xuống disk sau (async, checkpoint)

  * Crash recovery
    - Crash → đọc WAL từ checkpoint
    - Replay lại thay đổi → restore state đúng 

  * ACID trong PostgreSQL

  * Atomicity
    - Transaction commit/rollback
    - Nếu crash trước commit → bỏ toàn bộ
    - Nếu commit → redo từ WAL

  * Consistency
    - Constraint: PK, FK, UNIQUE, CHECK
    - Trigger / rule đảm bảo data hợp lệ
    - MVCC đảm bảo không thấy trạng thái “nửa commit”

  * Isolation
    - Dựa trên MVCC + snapshot
    - Mỗi transaction thấy version data tại thời điểm bắt đầu
    - Tránh:
      - dirty read
      - non-repeatable read (tuỳ isolation level)
    - Write conflict được serialize bằng row-level lock

  * Durability
    - COMMIT = WAL fsync thành công
    - Crash → replay WAL → không mất data đã commit
    - Full page write giúp tránh corrupted page khi crash

# Kafka (KRaft)
  * Architecture (tổng quan)
    * Cluster gồm:
      * Broker: xử lý data (produce/consume)
      * Controller: quản lý metadata
    * Metadata gồm:
      * topic, partition, ISR, config
    * KRaft = Kafka tự quản lý metadata (không cần ZooKeeper)

  * ZooKeeper mode (kiến trúc cũ)
    * Metadata lưu trong ZooKeeper
    * Kafka controller đọc/ghi metadata qua ZooKeeper 
    * Nhược điểm
      * Tăng độ phức tạp vận hành. Phải vận hành 2 hệ thống (Kafka + ZooKeeper)
      * Write bottleneck tại ZooKeeper leader do mọi write metadata phải qua leader
      * Khó scale khi metadata lớn
  * KRaft (Kafka Raft mode)
    * Thay ZooKeeper bằng Raft consensus
    * Metadata được quản lý trực tiếp trong Kafka 
    * Controller nodes tạo thành quorum

  * Metadata (cách lưu)
    * Metadata lưu dưới dạng **log append-only**
    * Lưu trong internal topic: `__cluster_metadata` 
    * Mỗi update = 1 record trong log
    * Replicate giữa các controller

  * Controller quorum
    * Nhóm controller node dùng Raft
    * 1 node là leader (active controller)
    * Leader:
      * xử lý metadata request
      * ghi metadata vào log
    * Followers:
      * replicate metadata
    * Failover nhanh (leader election qua Raft) 

  * Write flow (metadata)
    1. Broker gửi request (create topic, update config…)
    2. Controller leader nhận request
    3. Ghi record vào metadata log
    4. Replicate tới majority (quorum)
    5. Commit → metadata có hiệu lực
    6. Broker fetch metadata mới

  * Metadata sync (broker)
    * Broker giữ local copy của metadata
    * Khi start:
      * load metadata cache
      * fetch delta từ controller 
    * Metadata được replay → update state local

  * Ưu điểm KRaft
    * Simplicity
      * Không cần ZooKeeper → 1 system duy nhất 
    * Performance
      * Ít network hop hơn
      * Metadata propagation nhanh hơn 
    * Scalability
      * Scale tới millions partitions 
    * Consistency
      * Raft đảm bảo strong consistency
