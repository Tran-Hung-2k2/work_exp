# Chapter 4: Encoding and Evolution

## 1. Tổng quan

- Trong hệ thống thực tế:
    - Data không đứng yên → luôn thay đổi theo thời gian
    - Schema cũng phải evolve

→ Hai vấn đề chính:

1. **Encoding (serialization)** – cách biểu diễn data
2. **Evolution** – làm sao thay đổi schema mà không break hệ thống

---

## 2. Encoding Data (Serialization)

### 2.1 Mục tiêu

- Convert object → bytes (để:
    - lưu disk
    - gửi qua network)

---

### 2.2 Language-Specific Formats

Ví dụ:

- Java Serialization
- Python pickle

#### Ưu điểm:

- Dễ dùng

#### Nhược điểm:

- Không portable (phụ thuộc language)
- Security risk
- Khó evolve

→ Không phù hợp hệ thống distributed

---

### 2.3 Text Formats (JSON, XML, CSV)

#### JSON

- Phổ biến nhất

##### Ưu điểm:

- Readable
- Interoperable

##### Nhược điểm:

- Không có schema rõ ràng
- Không efficient (size lớn)
- Ambiguous (number, null, encoding)

---

### 2.4 Binary Formats

#### Ví dụ:

- MessagePack
- BSON

#### Ưu điểm:

- Nhỏ hơn JSON
- Faster parse

---

## 3. Schema-based Encoding

### 3.1 Thư viện phổ biến

- Thrift
- Protocol Buffers (Protobuf)
- Avro

---

### 3.2 Ưu điểm

- Có schema rõ ràng
- Compact
- Forward & backward compatible

---

### 3.3 Cách hoạt động

- Schema define structure
- Data encode theo schema

→ Reader cần schema để decode

---

### 3.4 So sánh

| Format   | Schema | Readable | Efficient |
| -------- | ------ | -------- | --------- |
| JSON     | ❌     | ✔        | ❌        |
| Protobuf | ✔      | ❌       | ✔         |
| Avro     | ✔      | ❌       | ✔         |

---

## 4. Schema Evolution

### 4.1 Vấn đề

- Code mới + code cũ cùng tồn tại
- Data cũ + data mới cùng tồn tại

---

### 4.2 Compatibility

#### Backward compatibility

- Code mới đọc được data cũ

#### Forward compatibility

- Code cũ đọc được data mới

---

### 4.3 Quy tắc thiết kế

- Không xóa field bắt buộc
- Chỉ thêm field optional
- Có default value
- Không reuse field ID

---

## 5. Dataflow

### 5.1 Dataflow Through Databases

- App A write DB
- App B read DB

→ DB đóng vai trò trung gian

---

### 5.2 Dataflow Through Services

#### REST / RPC

- Client → Server → Response

##### Ưu điểm:

- Simple

##### Nhược điểm:

- Tight coupling
- Versioning khó

---

### 5.3 Message Passing (Async)

- Producer → Queue → Consumer

#### Ưu điểm:

- Decoupling
- Scalable

#### Nhược điểm:

- Debug khó
- Eventual consistency

---

## 6. RPC vs REST

### RPC

- Gọi function từ xa
- Giống local call

#### Vấn đề:

- Network không reliable
- Timeout, retry phức tạp

---

### REST

- Resource-based
- Stateless

#### Ưu điểm:

- Simple
- Web-friendly

---

## 7. Message Queues & Event Streams

### 7.1 Message Queue

- RabbitMQ, etc.

### 7.2 Event Streaming

- Kafka

---

### 7.3 Ưu điểm:

- Loose coupling
- Replay data
- Scalable

---

## 8. Evolution in Distributed Systems

### 8.1 Challenge

- Nhiều service version khác nhau

---

### 8.2 Giải pháp:

- Versioning API
- Backward compatibility
- Schema registry (Kafka + Avro)

---

## 9. Key Insights

- Data format ảnh hưởng:
    - Performance
    - Compatibility
    - Scalability

- Schema evolution là bắt buộc
- Binary formats (Protobuf, Avro) rất quan trọng trong production
- Event-driven architecture giúp decouple system
