# Chapter 3: Storage and Retrieval

## 1. Tổng quan

- Database = hệ thống:
    - **Store data**
    - **Retrieve data efficiently**

→ Hiệu năng phụ thuộc mạnh vào:

- Data structure
- Indexing strategy
- Disk I/O pattern

---

## 2. Data Structures in Databases

### 2.1 Log-Structured Storage

- Append-only file (ghi tuần tự)
- Mỗi write = append vào log

#### Ưu điểm:

- Write nhanh (sequential I/O)
- Simple

#### Nhược điểm:

- File lớn dần → cần cleanup

---

### 2.2 Hash Index

- Key → offset trong file

#### Cách hoạt động:

- Store data trong file
- Hash table giữ vị trí key

#### Ưu điểm:

- Lookup O(1)
- Simple

#### Nhược điểm:

- Chỉ support exact match
- Không support range query

---

## 3. SSTables & LSM-Trees

### 3.1 SSTable (Sorted String Table)

- File immutable
- Key được **sort**

#### Ưu điểm:

- Efficient merge
- Binary search

---

### 3.2 LSM-Tree (Log-Structured Merge Tree)

#### Ý tưởng:

- Write vào memory (memtable)
- Flush xuống disk thành SSTable
- Background merge (compaction)

---

### 3.3 Ưu điểm:

- Write cực nhanh
- Scale tốt với write-heavy workload

---

### 3.4 Nhược điểm:

- Read chậm hơn (phải check nhiều level)
- Compaction tốn tài nguyên

---

## 4. B-Trees

### 4.1 Cấu trúc

- Tree cân bằng
- Node chứa nhiều keys

---

### 4.2 Đặc điểm:

- Data được lưu trực tiếp trong tree
- Disk-friendly (page-based)

---

### 4.3 Ưu điểm:

- Read nhanh
- Support range query tốt

---

### 4.4 Nhược điểm:

- Write chậm hơn LSM (random write)
- Fragmentation

---

## 5. B-Trees vs LSM-Trees

| Aspect      | B-Tree     | LSM-Tree    |
| ----------- | ---------- | ----------- |
| Write       | Chậm hơn   | Nhanh       |
| Read        | Nhanh      | Chậm hơn    |
| Range query | Tốt        | Tốt         |
| Disk I/O    | Random     | Sequential  |
| Use case    | Read-heavy | Write-heavy |

---

## 6. Other Index Structures

- Secondary index
- Full-text index
- In-memory index

→ Trade-off giữa:

- Speed
- Memory
- Complexity

---

## 7. OLTP vs OLAP

### 7.1 OLTP (Online Transaction Processing)

- Nhiều transaction nhỏ
- Low latency
- Ví dụ: banking, ecommerce

---

### 7.2 OLAP (Analytics)

- Query lớn
- Scan nhiều data
- Ví dụ: BI, report

---

## 8. Data Warehousing

- Separate OLTP và OLAP

### Pipeline:

1. Extract
2. Transform
3. Load (ETL)

---

### Schema phổ biến:

- Star schema
- Snowflake schema

---

## 9. Column-Oriented Storage

### 9.1 Ý tưởng:

- Store theo column thay vì row

---

### 9.2 Ưu điểm:

- Query nhanh cho analytics
- Chỉ đọc column cần thiết

---

### 9.3 Compression

- Column giống nhau → compress tốt

---

### 9.4 Nhược điểm:

- Write phức tạp hơn
- Không phù hợp OLTP

---

## 10. Materialized Views & Data Cubes

### 10.1 Materialized View

- Precompute query result

#### Ưu điểm:

- Query cực nhanh

#### Nhược điểm:

- Update phức tạp

---

### 10.2 Data Cube

- Aggregate multi-dimension

Ví dụ:

- Sales theo:
    - Time
    - Location
    - Product

---

## 11. Key Insights

- Storage engine = core của database
- Không có cấu trúc nào tối ưu cho mọi case
- LSM-tree đang phổ biến trong hệ thống lớn
- B-tree vẫn rất quan trọng (MySQL, PostgreSQL)
- OLTP và OLAP cần thiết kế khác nhau

---

## 12. Kết luận

> Hiệu năng database phụ thuộc vào:
>
> - Data structure (B-tree vs LSM)
> - Access pattern (read vs write)
> - Workload (OLTP vs OLAP)

→ Chọn sai storage engine = bottleneck lớn
