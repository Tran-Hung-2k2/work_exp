# Chapter 1: Reliable, Scalable, and Maintainable Applications

## 1. Tổng quan

- Ứng dụng hiện đại thường là **data-intensive**:
    - Vấn đề chính: dữ liệu (volume, complexity, velocity), không phải CPU.
- Các thành phần phổ biến:
    - Database (lưu trữ)
    - Cache (tăng tốc đọc)
    - Search index (tìm kiếm)
    - Stream processing (xử lý async)
    - Batch processing (xử lý offline)

→ Một hệ thống thực tế thường là **tổ hợp nhiều công cụ** → cần tư duy như một **data system designer**.

---

## 2. Ba mục tiêu cốt lõi

### 2.1 Reliability (Độ tin cậy)

**Định nghĩa:** Hệ thống vẫn hoạt động đúng ngay cả khi có lỗi.

#### Các loại lỗi:

- **Hardware faults**:
    - Disk hỏng, mất điện, lỗi RAM
    - Giải pháp: redundancy (RAID, replication)

- **Software errors**:
    - Bug, memory leak, dependency failure
    - Khó dự đoán hơn vì có thể xảy ra đồng loạt

- **Human errors**:
    - Sai config, thao tác nhầm
    - Thực tế là nguyên nhân chính gây downtime

#### Cách cải thiện reliability:

- Thiết kế hệ thống chịu lỗi (fault-tolerant)
- Testing (unit → integration)
- Monitoring & observability
- Rollback nhanh
- Sandbox môi trường test
- Chaos engineering (test lỗi chủ động)

---

### 2.2 Scalability (Khả năng mở rộng)

**Định nghĩa:** Hệ thống xử lý tốt khi load tăng.

#### 2.2.1 Mô tả load

- Requests/sec
- Read/write ratio
- Active users
- Data volume

**Ví dụ Twitter:**

- Write (tweet): ~4.6k/sec
- Read (timeline): ~300k/sec
- Vấn đề chính: **fan-out**

#### 2.2.2 Hai cách thiết kế timeline

1. **Pull model**:
    - Query tweets khi đọc
    - Đơn giản nhưng chậm

2. **Push model**:
    - Precompute timeline khi viết
    - Nhanh khi đọc, tốn chi phí write

→ Thực tế: hybrid (celebrity dùng pull)

---

#### 2.2.3 Đo performance

- **Throughput**: số lượng xử lý / giây
- **Response time**: thời gian phản hồi

**Quan trọng: dùng percentiles, không dùng average**

- p50 (median)
- p95, p99, p999 (tail latency)

→ Tail latency ảnh hưởng mạnh đến UX

---

#### 2.2.4 Scale như thế nào?

- **Scale up** (vertical):
    - Máy mạnh hơn
- **Scale out** (horizontal):
    - Nhiều máy hơn (distributed)

→ Thực tế: kết hợp cả hai

---

### 2.3 Maintainability (Khả năng bảo trì)

#### Ba yếu tố chính:

### a. Operability

- Giúp vận hành dễ dàng:
    - Monitoring tốt
    - Automation
    - Deploy dễ
    - Debug nhanh

---

### b. Simplicity

- Giảm độ phức tạp:
    - Tránh "big ball of mud"
    - Clear abstraction
    - Loose coupling

---

### c. Evolvability

- Dễ thay đổi theo yêu cầu mới:
    - Thiết kế linh hoạt
    - Schema evolution
    - Modular architecture

---

## 3. Key Insights

- Không có hệ thống nào hoàn hảo → luôn trade-off
- Reliability, Scalability, Maintainability là nền tảng design
- Distributed systems = phức tạp hơn rất nhiều
- Architecture phải dựa trên:
    - Load pattern
    - Use case cụ thể
    - Trade-offs
