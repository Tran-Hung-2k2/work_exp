# Chapter 2: Data Models and Query Languages

## 1. Tổng quan

- Data model = cách tổ chức và biểu diễn dữ liệu
- Là abstraction quan trọng nhất trong hệ thống dữ liệu

→ Quyết định:

- Cách lưu trữ
- Cách query
- Performance & scalability

---

## 2. Relational Model vs Document Model

### 2.1 Relational Model (SQL)

- Dữ liệu lưu dưới dạng **table (rows + columns)**
- Schema rõ ràng
- Quan hệ thông qua **foreign key**

#### Ưu điểm:

- Strong consistency
- Query mạnh (JOIN)
- Chuẩn hóa (normalization)

#### Nhược điểm:

- Khó handle dữ liệu nested
- Object-relational mismatch

---

### 2.2 Document Model (NoSQL)

- Dữ liệu lưu dạng **JSON / BSON**
- Flexible schema (schema-less)

#### Ưu điểm:

- Tự nhiên với object (nested structure)
- Dễ scale (horizontal)
- Ít JOIN → performance tốt hơn

#### Nhược điểm:

- Khó join phức tạp
- Dễ duplicate data

---

### 2.3 Khi nào dùng cái nào?

| Use case                   | Nên dùng   |
| -------------------------- | ---------- |
| Data có quan hệ phức tạp   | Relational |
| Data dạng tree / JSON      | Document   |
| Cần join nhiều             | Relational |
| Scale lớn, flexible schema | Document   |

---

## 3. Object-Relational Impedance Mismatch

- Code (OOP) vs Database (Relational) khác nhau

Ví dụ:

- Object có nested structure
- DB phải split thành nhiều bảng

→ Giải pháp:

- ORM (Hibernate, etc.)
- Nhưng:
    - Có overhead
    - Không giải quyết triệt để

→ Document DB giải quyết tốt hơn vì:

- Lưu object trực tiếp

---

## 4. Relationships

### 4.1 Many-to-One

- Ví dụ:
    - user → country
- Relational:
    - dùng foreign key
- Document:
    - store trực tiếp hoặc reference

---

### 4.2 Many-to-Many

- Ví dụ:
    - user ↔ group

→ Relational:

- join table

→ Document:

- embed hoặc reference

---

### 4.3 Trade-off

- Normalization vs Denormalization

| Strategy    | Ưu điểm     | Nhược điểm     |
| ----------- | ----------- | -------------- |
| Normalize   | Consistency | Join cost      |
| Denormalize | Fast read   | Duplicate data |

---

## 5. Query Languages

### 5.1 Imperative vs Declarative

#### Imperative

- Mô tả cách làm
- Ví dụ: loop từng record

#### Declarative (SQL)

- Mô tả _what_, không phải _how_
- DB optimizer quyết định execution

→ Declarative tốt hơn:

- Dễ optimize
- Dễ parallelize

---

### 5.2 MapReduce

- Mô hình xử lý dữ liệu:
    - Map → transform
    - Reduce → aggregate

#### Ưu điểm:

- Scale tốt
- Phù hợp distributed system

#### Nhược điểm:

- Verbose
- Không trực quan như SQL

---

## 6. Graph Data Models

### 6.1 Khi nào cần graph?

- Data có nhiều quan hệ phức tạp:
    - Social network
    - Recommendation
    - Routing

---

### 6.2 Property Graph Model

- Node (entity)
- Edge (relationship)
- Property (metadata)

---

### 6.3 Query Graph

#### Cypher (Neo4j)

- Query bằng pattern matching

#### SQL (có thể nhưng khó)

#### SPARQL (triple store)

---

### 6.4 Ưu điểm Graph

- Expressive cho relationship
- Query path rất mạnh

---

## 7. Schema & Flexibility

### 7.1 Schema-on-write (Relational)

- Validate khi ghi
- Strict

### 7.2 Schema-on-read (NoSQL)

- Validate khi đọc
- Flexible

---

### 7.2 Trade-off

| Schema          | Ưu điểm     | Nhược điểm     |
| --------------- | ----------- | -------------- |
| Fixed schema    | Data clean  | Hard to evolve |
| Flexible schema | Adapt nhanh | Data messy     |

---

## 8. Key Insights

- Không có data model tốt nhất → phụ thuộc use case
- Relational vẫn rất mạnh (không lỗi thời)
- NoSQL không thay thế SQL → bổ sung
- Graph model cực mạnh cho relationship-heavy data
- Declarative query = xu hướng chính

---

## 9. Kết luận

> Data model quyết định:
>
> - Cách bạn nghĩ về dữ liệu
> - Cách hệ thống scale
